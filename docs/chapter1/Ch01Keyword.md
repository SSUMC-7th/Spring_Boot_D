✨ 1주차 keyword ✨
=============

✅ # 외래키
  외래키는 테이블 간 관계를 나타낼 때 사용하며, 다른 테이블의 기본키를 참조해 외래키로 사용한다.
  또한 하나의 테이블에 여러 외래키가 존재할 수 있다.
  외래키는 기본키에 반해 NULL 값을 가질 수 있다.
  만약 NULL 값이 아니라면, 반드시 참조하는 테이블의 기본키 중 하나를 가져야 한다.
  이는 곧 테이블 간의 참조 데이터 무결성을 위한 제약 조건이기도 하다.

  -----

✅ # 기본키
  주 키, primary key 등으로 불리며 후보키들 중 메인으로 선정되는 키이다.
  주로 ID나 번호로 사용되며, AUTO_INCREMENT를 통해 자동으로 값이 증가하도록 설정할 수 있다.
  기본적으로 슈퍼키 안에 후보키가 있고, 후보키는 기본키와 대체키로 구성되어 있다.
  따라서 기본키는 후보키의 성질을 가지며 유일성과 최소성을 충족하기 때문에,
  레코드를 식별할 때 기준이 되는 반드시 필요한 키이다.

  기본키가 가져야 하는 조건은 다음과 같다.

  * 값의 변동이 잦은 후보키는 기본키로 부적절하다.
  * NULL 값을 가질 수 있는 속성이 포함되는 후보키는 부적절하다.
  * 후보키 중 단순한 키를 기본키로 선택한다.
  * 하나의 테이블에는 반드시 하나의 기본키만 존재한다.

  이러한 조건은 곧 개체 무결성을 만족하기 위한 제약 조건이기도 하다.

  -----

✅ # ER 다이어그램
  개체-관계(E-R) 모델은 현실 세계에 존재하는 데이터와 그들 간의 관계를
  사람이 이해할 수 있는 형태로 명확하게 표현하기 위한 모델이고, 이를 시각적으로 표현한 것이 ER 다이어그램이다.
  ER 다이어그램은 주로 데이터베이스 설계의 초기 단계에서 개념적 모델을 만드는 데 사용된다.
  ER 다이어그램의 구성요소는 개체, 관계, 속성으로 구성된다. 

  * 개체 : 개체는 정의 가능한 사물 또는 개념을 의미하며, 유무형의 정보를 가질 수 있는 요소이다.
           개체의 이름은 단수형이고 유일하며, 모든 개체는 하나 이상의 식별자(기본키)를 가져야 한다.
           개체는 데이터베이스의 물리적 설계 단계에서 테이블로 변환되는 특징이 있다.
    
  * 속성 : 개체가 갖는 속성을 의미하며, 개체의 이름과 속성의 이름이 중복되면 안 된다.
           속성은 데이터베이스의 물리적 설계 단계에서 컬럼으로 변환되는 특징이 있다.
    
  * 관계 : 두 개 이상의 개체 사이에 존재하는 연관성으로, 두 개체 간에 선을 긋고 관계 명칭을 기재한다.
           이때 점선은 선택적인 사항을, 실선은 필수적인 사항을 의미한다.
           또한 관계는 One to One(1:1), One to Many(1:N), Many to Many(N:M)로 카디널리티를 표현할 수 있다.
           데이터 모델링에서는 두 개체 간의 관계가 N:M으로 매칭되는 경우, 완성되지 않은 모델로 간주하기 때문에
           두 개체의 관계를 1:N, 1:M으로 조정하는 작업이 필요하다.
           이를 위해 두 개체 사이에 또 다른 개체를 추가하여 하나의 N:M 관계를 두 개의 M:1 관계로 전환할 수 있다.

  -----
  
✅ # 복합키
  복합키는 두 개 이상의 컬럼을 묶어서 하나의 기본키로 지정하는 것을 말한다.
  주로 하나의 컬럼 만으로 유일성을 보장할 수 없는 경우에 사용된다.
  만약 기본키가 복합키라면, 복합키 또한 당연히 유일성과 최소성을 만족해야 한다.

  -----

✅ # 연관관계
  연관관계는 두 개 이상의 사물이 서로 관련된 상태를 표현하는 관계로, 방향, 다중성, 연관관계 주인이 있다.
  이때 사물 사이는 실선으로 연결하여 표현하며 방향성은 화살표로 표현한다.

  * 방향 : 단방향과 양방향이 존재한다. 단방향 연관관계는 한 개체가 다른 개체를 참조할 수 있지만,
           반대 방향으로는 참조할 수 없는 관계이다.
           단방향 연관관계를 양쪽으로 설정함으로써 양방향 연관관계를 만들 수 있다.
           서로에게 영향을 주는 양방향 연관관계의 경우 화살표를 생략하고 실선으로만 연결한다.
    
  * 다중성 : 다대일(N:1), 일대다(1:N), 일대일(1:1), 다대다(N:M) 연관관계가 존재한다.
            일반적으로 관계형 데이터베이스에서는 다대다 연관관계를 다룰 수 없기 때문에,
            중간에 연결 테이블을 추가하여 일대다, 다대일 관계로 풀어내야 한다.
    
    > 관련 어노테이션 : @ManyToOne, @OneToMany, @OneToOne
    
  * 연관관계의 주인 : 양방향 연관관계를 설정하면 한 쪽을 연관관계의 주인으로 설정해야 하며, 주인이 외래 키를 관리한다.
                     일대다(1:N) 혹은 다대일(N:1) 관계에서 외래키는 항상 “다” 쪽에 존재한다.
                     따라서 객체의 양방향 관계에서 연관 관계 주인은 항상 “다” 쪽이다.
                     반면에 일대일(1:1) 관계에서는 둘 중 어느 테이블에서도 외래키를 가질 수 있다.
                     따라서 두 개체 중 누가 외래키를 가질 것인지 선택해야 하는데, 보통 접근횟수가 많은 개체를 주 테이블로 설정한다.
    
  주로 다대일(N:1) 연관관계가 가장 많이 사용된다.

✅ # 정규화
  정규화는 관계형 데이터 모델에서 데이터의 중복성을 제거하여 이상 현상을 방지하고,
  데이터의 일관성과 정확성을 유지하기 위해 무손실 분해하는 과정이다.
  이때 데이터의 중복성과 삽입/삭제/갱신 이상을 제거하는 것이 중요하다.

  릴레이션 조작 시 데이터가 불필요하게 중복되어 예기치 않게 발생하는 이상 현상에는 크게 3가지 종류가 있다.

  * 삽입 이상 : 릴레이션에 데이터를 삽입할 때 의도와는 상관없이 원하지 않은 값들도 함께 삽입되는 현상
  * 삭제 이상 : 릴레이션에서 한 튜플을 삭제할 때 의도와는 상관없이 원하지 않은 값들도 함께 삽입되는 현상
  * 갱신 이상 : 릴레이션에서 튜플에 있는 속성값을 갱신할 때 일부 튜플의 정보만 갱신되어 정보의 모순이 생기는 현상

  이러한 이상 현상이 발생하지 않도록 데이터베이스를 6단계에 걸쳐 정규화 할 수 있다.

  1. 1정규형(1NF) : 테이블 내의 속성 값이 원자값으로 구성되도록 분리
  2. 2정규형(2NF) : 부분 함수 종속 제거(완전 함수적 종속 관계)
  3. 3정규형(3NF) : 이행 함수 종속 제거(A→B, B→C, A→C 관계 분리)
  4. 보이스-코드 정규형(BCNF) : 결정자가 후보 키가 아닌 함수 종속 제거
  5. 4정규형(4NF) : 다치(다중 값) 종속성 제거
  6. 5정규형(5NF) : 조인 종속성 제거

  실무에서는 3차 정규화까지만 자주 활용된다.
  이러한 정규화를 통해 데이터 구조의 안정성을 최대화하고, 어떠한 릴레이션도 데이터베이스 내에서 표현할 수 있게 된다.

  -----

✅ # 반정규화
  반 정규화는 정규화된 개체, 속성, 관계에 대해 성능 향상과 개발 운영의 단순화를 위해
  중복, 통합, 분리 등을 수행하는 데이터 모델링의 기법이다.

  반 정규화를 통해 성능 향상과 관리의 효율성이 증가할 수 있지만,
  중복 제거를 위해 테이블을 분리했던 과정을 역으로 수행하다보니 데이터의 일관성과 정합성이 저하될 수 있다.
  또한 유지를 위한 비용이 별도로 발생하여 성능에 나쁜 영향을 미칠 수 있다.

  반 정규화 기법에는 다음과 같은 것들이 있다.

  * 테이블 병합 : 일대일, 일대다 관계를 통합하여 조인 횟수를 줄여 성능을 향상하는 기법.
                 슈퍼타입/서브타입 테이블을 통합함으로써 성능 향상을 도모할 수 있다.
    
  * 테이블 분할 : 수평 분할(레코드를 기준으로 테이블 분할),
                 수직 분할(하나의 테이블이 가지는 컬럼의 개수가 증가하는 경우 활용.갱신 위주의 속성 분할.)
    
  * 중복 테이블 추가 : 대량의 데이터들에 대한 집계함수를 사용하여 실시간 통계정보를 계산하는 경우에
                      효과적인 수행을 위해 별도의 통계 테이블을 두거나 중복 테이블을 추가
    
  * 컬럼 중복화 : 조인 성능 향상을 위한 중복 허용
    
  * 중복관계 추가 : 데이터를 처리하기 위한 여러 경로를 거쳐 조인이 가능하지만
                   이때 발생할 수 있는 성능 저하를 예방하기 위해 추가적 관계를 맺는 방법
